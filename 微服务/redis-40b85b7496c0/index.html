Redis(三)—秒杀场景，redis 如何应对并发

并发

秒杀是一个非常典型的活动场景，在 618、双 12 等电商促销活动中，都会有秒杀场景，这种场景的业务特点是 系统需要瞬时的处理大量高并发请求，而对促销的商品需要限时限量。

在秒杀场景中带来的一些问题：

  1. 如何生成全局唯一 ID
  2. 库存超卖问题
  3. 一人一单问题
  4. 异步秒杀问题

下面我会一一说明。

1. 全局唯一 ID

全局唯一 ID 并不只是在秒杀场景中，在一般的分布式系统中，对于全局唯一 ID 也有一定的要求，但是在秒杀场景，我们需要设计出性能更好、更加安全可靠的全局唯一 ID，所以在这里单独提出来。

1.1 数据库自增的问题

对于 MySQL 数据库表，主键 ID 自增策略就是一种唯一 ID，但作为分布式系统下的全局唯一 ID，多少还是有点不够格。

  1. 自增规律明显，在一些电商项目，通常都会有订单表，如果使用数据库主键自增的策略，那么生成的 ID 具有太明显的规律，用户可以很容易的猜出一些敏感信息，比如商品在一天中卖了多少单等。
  2. 并且 MySQL 的单表容量不宜超过 500W，当项目运营的越来越久，数据量过大之后，我们会进行分库分表处理，但这样处理之后，依赖单表主键自增显然没有办法保证 ID 的唯一性，就需要使用额外的策略，得不偿失。

1.2 生成全局唯一 ID

想要在分布式系统中生成全局唯一 ID，就需要使用合适的全局唯一 ID 生成器，其一般要满足下面的特性：

  1. 唯一性，必须要保证的、最基本的特征
  2. 高可用
  3. 高性能
  4. 递增性，这里强调趋势递增，类似 MySQL 的数据存储是基于 B+ 树的聚簇索引，乱序的 ID 势必导致写入性能受到影响。
  5. 安全性，如果 ID 可以被简单的枚举出来，那么就有潜在的数据安全风险，如上面描述的电商订单表。

这里我们采用 redis 来生成全局唯一 ID。

主要是依赖于 redis 的原子操作 incr 来完成的，同时 redis 还可以做日切流水号，比如订单号 = 日期 + 当日增长号，即每天在 redis 中生成一个 key，使用 incr 进行累加，为了 ID 的安全性，也可以不直接使用 redis 自增的数值，可以拼接一些其他的信息。
