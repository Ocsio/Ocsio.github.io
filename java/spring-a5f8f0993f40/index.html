SpringBoot — 自定义 Starter

SpringBoot — 自定义 starter

1. starter 介绍

SpringBoot 在配置上相比 Spring 要简单许多，其核心在于 spring-boot-starter，在使用 SpringBoot 来搭建一个项目时，只需要引入官方提供的 starter，就可以直接使用，免去了各种配置。starter 简单来讲就是引入了一些相关依赖和一些初始化的配置。

Spring 官方提供了很多 starter，第三方也可以定义 starter。为了加以区分，starter 从名称上进行了如下规范：

  • Spring 官方提供的 starter 名称为：spring-boot-starter-xxx，例如 Spring官方提供的 spring-boot-starter-web
  • 第三方提供的 starter 名称为：xxx-spring-boot-starter，例如 mybatis 提供的 mybatis-spring-boot-starter

2. starter 原理

SpringBoot 之所以可以帮助简化项目的搭建和开发过程，主要是基于它提供的起步依赖和自动配置。

2.1 起步依赖

起步依赖，就是将具备某种功能的坐标打包到一起，可以简化依赖导入的过程。比如，导入 spring-boot-starter-web，那么和 web 开发相关的 jar 包都一起导入到项目中了。如下图所示：

2.2 自动配置

自动配置，就是无须手动配置 xml，自动配置并管理 bean，简化开发过程。SpringBoot是如何完成自动配置的呢？

自动配置涉及到如下几个关键步骤：

  • 基于 Java 代码的 Bean 配置
  • 自动配置条件依赖
  • Bean 参数获取
  • Bean 的发现
  • Bean 的加载

以 mybatis-spring-boot-starter 说明。

这些注解是 SpringBoot 特有的，常见的条件依赖注解有：

注解 功能说明
@ConditionalOnBean 仅在当前上下文中存在某个 bean 时，才会实例化该 bean
@ConditionalOnClass 某个 class 位于类路径上，才会实例化该 bean
@ConditionalOnExpression 当表达式为 true 的时候，才会实例化该 bean
@ConditionalOnProperty 仅在配置文件中存在相应的配置，才会实例化该 bean
@ConditionalOnSingleCandidate 仅当在上下文中存在某个单例 bean 时，才会实例化该 bean
@ConditionalOnMissingBean 仅在当前上下文中不存在某个 bean 时，才会实例化 bean
@ConditionalOnMissingClass 某个 class 在类路径上不存在的时候，才会实例化该 bean
@ConditionalOnNotWebApplication 不是 web 应用时才会实例化该 bean
@AutoConfigureAfter 在某个 bean 完成自动配置后实例化该 bean
@AutoConfigureBefore 在某个 bean 完成自动配置前实例化该 bean

Bean 的加载

在 SpringBoot 应用中要让一个普通类交给 Spring 容器管理，通常有以下方法：

  1. 使用 @Configuration 与 @Bean 注解
  2. 使用 @Component 及其衍生注解标注该类并且启用 @ComponentScan 自动扫描
  3. 使用 @Import 方法

其中 SpringBoot 实现自动配置使用的是 @Import 注解，AutoConfigurationImportSelector 类的 selectImports 方法返回一组从 META-INF/spring.factories 文件中读取的 bean 的全类名，这样 SpringBoot 就可以加载到这些 bean 并完成实例的创建工作。
